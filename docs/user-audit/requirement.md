# 用户注册审核 - 需求文档

## 功能概述

在现有用户体系基础上，新增注册审核机制。用户通过 App 端注册后，账号进入「待审核」状态，需由后台管理员审核通过后方可登录使用。审核未通过的用户将收到拒绝原因提示。

**业务价值：**
- 后台可控的用户准入机制，保障平台用户质量
- 管理员可查看注册用户信息并做出审核决定
- App 端用户注册后即时看到审核状态，体验清晰
- 审核拒绝后支持重新注册，降低用户流失

## 用户故事

- 作为未注册用户，我想注册后看到审核状态页面，以便知道我的账号正在等待审核。
- 作为待审核用户，我想在审核状态页面刷新查看最新状态，以便及时得知审核结果。
- 作为被拒绝的用户，我想看到拒绝原因，以便了解未通过的原因并重新注册。
- 作为审核通过的用户，我想在审核状态页面看到「去登录」入口，以便快速登录。
- 作为管理员，我想在后台查看待审核的用户列表，以便审核新注册用户。
- 作为管理员，我想审核通过或拒绝用户，并填写拒绝原因，以便管理用户准入。

## 功能列表

| 优先级 | 功能 | 描述 |
|--------|------|------|
| P0 | 注册状态变更 | 注册后用户状态从「直接启用」改为「待审核」，不再自动登录 |
| P0 | 登录审核校验 | 登录时校验审核状态，未通过审核的账号禁止登录 |
| P0 | 审核状态查询 | App 端通过手机号查询当前审核状态（免登录） |
| P0 | App 审核状态页 | 注册成功后跳转审核状态页，支持刷新查看结果 |
| P0 | 管理员审核接口 | 后台审核通过/拒绝用户，支持填写拒绝原因 |
| P1 | 管理端用户审核页 | 管理端新增 App 用户管理页面，支持按审核状态筛选 |
| P1 | 被拒绝后重新注册 | 审核被拒绝的用户可重新注册（覆盖旧记录） |

## 非功能性需求

### 安全
- 审核状态查询接口为免登录接口，仅返回审核状态和拒绝原因，不泄露用户其他敏感信息
- 审核操作需管理员权限（`/api/admin/**` 拦截器保护）
- 审核拒绝原因不对外暴露手机号等隐私字段

### 性能
- 审核状态查询接口响应时间 < 200ms
- 审核操作接口响应时间 < 500ms

### 兼容性
- 后端兼容现有 `app_user` 表结构，仅新增 `audit_status`、`audit_remark`、`audit_time` 字段
- 现有已注册用户默认审核状态为「已通过」（`audit_status = 1`）
- App 端兼容现有 DioClient / TokenManager / ApiInterceptor 架构
- 管理端兼容现有 AdminLayout / Router / React Query 架构

## 边界条件与异常场景

| 场景 | 触发条件 | 预期行为 |
|------|----------|----------|
| 待审核用户登录 | 登录时 audit_status = 0 | 返回错误码 10017「账号审核中，请耐心等待管理员审核」|
| 审核拒绝用户登录 | 登录时 audit_status = 2 | 返回错误码 10018「注册审核未通过」+ 拒绝原因 |
| 被拒绝用户重新注册 | 注册时手机号已存在且 audit_status = 2 | 允许重新注册，覆盖旧记录，重置为待审核状态 |
| 已通过用户重复注册 | 注册时手机号已存在且 audit_status = 1 | 返回错误码 10011「手机号已注册」|
| 待审核用户重复注册 | 注册时手机号已存在且 audit_status = 0 | 返回错误码 10011「手机号已注册」|
| 查询审核状态-手机号未注册 | 查询的手机号不存在 | 返回错误码 10010「手机号未注册」|
| 管理员重复审核 | 已审核通过的用户再次审核 | 返回错误码 10019「该用户已审核，无需重复操作」|
| 管理员审核不存在用户 | 审核的用户 ID 不存在 | 返回错误码 10002「用户不存在」|
