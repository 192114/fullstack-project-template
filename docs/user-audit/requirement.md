# 用户注册审核 - 需求文档

## 功能概述

在现有的用户注册流程中引入后台审核机制。用户注册后不能直接登录，需等待管理员审核通过后方可正常使用。审核驳回后，用户可修改注册信息重新提交审核。该功能覆盖 App 端（Flutter）和管理端（React Admin）两端。

## 用户故事

- 作为 App 用户，我希望注册后能查看审核进度，以便了解账号当前状态。
- 作为 App 用户，我希望审核驳回后能查看驳回原因，以便针对性修改后重新提交。
- 作为 App 用户，我希望登录时如果账号未通过审核，能获得明确的提示并跳转到审核状态页。
- 作为管理员，我希望能在后台查看所有待审核的用户列表，以便逐一审批。
- 作为管理员，我希望审核时能通过或驳回，并填写驳回原因，以便用户了解未通过的具体原因。

## 功能列表

| 优先级 | 功能 | 描述 |
|--------|------|------|
| P0 | 注册流程改造 | 注册成功后不再返回 Token，用户进入待审核状态（auditStatus=0） |
| P0 | 登录审核校验 | 登录时校验审核状态，未通过时返回特定错误码引导跳转 |
| P0 | 审核状态查询 | App 端免登录查询审核状态（含驳回原因） |
| P0 | 审核状态页 | App 端审核状态展示页，支持审核中/已通过/已驳回三种状态 |
| P0 | 管理端审核操作 | 管理端用户列表页支持审核通过/驳回（含驳回原因） |
| P1 | 驳回后重新提交 | 审核驳回后用户可修改注册信息（昵称、密码）重新发起审核 |
| P1 | 管理端审核筛选 | 管理端用户列表支持按审核状态筛选 |

## 非功能性需求

- 安全: 审核状态查询接口免登录但仅返回脱敏信息（手机号掩码）；重新提交需验证短信验证码
- 性能: 审核状态查询接口需支持高频轮询，考虑 Redis 缓存（后续优化）
- 兼容性: 现有已注册用户默认 auditStatus=1（已通过），不受审核流程影响
- 一致性: 注册、登录、重新提交三个入口的审核状态校验逻辑保持一致

## 边界条件与异常场景

| 场景 | 触发条件 | 预期行为 |
|------|----------|----------|
| 已注册手机号再次注册 | 手机号存在且 auditStatus=0 或 1 | 返回错误「手机号已注册」 |
| 驳回后重新注册 | 手机号存在且 auditStatus=2 | 允许覆盖更新，重置审核状态为待审核 |
| 登录时待审核 | auditStatus=0 | 返回错误码 10017，提示「账号审核中」 |
| 登录时已驳回 | auditStatus=2 | 返回错误码 10018，提示「注册审核未通过」并附带驳回原因 |
| 重复审核 | 对已审核用户再次操作 | 返回错误码 10019，提示「该用户已审核」 |
| 重新提交时非驳回状态 | auditStatus=0 或 1 | 返回错误，不允许重新提交 |
| 审核状态查询手机号不存在 | 查询的手机号未注册 | 返回错误码 10010，提示「手机号未注册」 |
