# 用户体系 - 需求文档

## 功能概述

实现完整的用户体系，以手机号为核心身份标识，支持密码登录和短信验证码登录两种方式。采用双 Token 认证机制（Access Token + Refresh Token），兼顾安全性与用户体验。

**业务价值：**
- 手机号作为唯一身份标识，降低注册门槛
- 双登录方式（密码 / 验证码）满足不同场景需求
- 双 Token 机制实现无感续期，提升用户体验
- 完善的个人中心管理功能

## 用户故事

- 作为未注册用户，我想通过手机号 + 验证码注册账号，以便快速成为平台用户。
- 作为已注册用户，我想通过手机号 + 密码登录，以便快速访问我的账号。
- 作为已注册用户，我想通过手机号 + 验证码登录，以便在忘记密码时也能登录。
- 作为已登录用户，我想在 Access Token 过期后自动续期，以便无需重新登录。
- 作为已登录用户，我想查看和编辑我的个人资料，以便维护我的账号信息。
- 作为已登录用户，我想修改密码，以便保障账号安全。
- 作为忘记密码的用户，我想通过手机验证码重置密码，以便恢复账号访问。
- 作为已登录用户，我想退出登录，以便在公共设备上保护我的账号安全。

## 非功能性需求

### 安全
- 密码使用 Argon2 算法加密存储，不存储明文
- 短信验证码 6 位数字，有效期 5 分钟，验证后立即失效
- 同一手机号发送验证码间隔不少于 60 秒（防刷）
- Refresh Token 一次性使用，刷新后生成新 Token（Token 轮转）
- 登录失败不区分「手机号未注册」和「密码错误」（防信息泄露）
- Access Token 通过 Sa-Token 管理，存储于 Redis
- 密码长度 6-64 位，不限制字符类型

### 性能
- 短信验证码发送接口响应时间 < 2s（含短信网关调用）
- 登录接口响应时间 < 500ms
- Token 刷新接口响应时间 < 200ms
- Refresh Token 存储于 Redis，TTL 7 天

### 兼容性
- 后端兼容现有 `app_user` 表结构，新增 `phone`、`avatar` 字段
- 前端兼容现有 DioClient / TokenManager / ApiInterceptor 架构
- 保留现有用户管理 CRUD 接口（管理端使用）

## 边界条件与异常场景

| 场景 | 触发条件 | 预期行为 |
|------|----------|----------|
| 手机号已注册 | 注册时手机号已存在 | 返回错误码 10011「手机号已注册」|
| 手机号未注册 | 验证码登录时手机号不存在 | 返回错误码 10010「手机号未注册」|
| 密码错误 | 密码登录时密码不匹配 | 返回错误码 10004「手机号或密码错误」|
| 账号被禁用 | 登录时 status = 0 | 返回错误码 10003「账号已被禁用」|
| 验证码过期 | 验证码超过 5 分钟 | 返回错误码 10012「验证码已过期，请重新获取」|
| 验证码错误 | 输入的验证码不匹配 | 返回错误码 10013「验证码错误」|
| 验证码发送频繁 | 60 秒内重复发送 | 返回错误码 10014「验证码发送过于频繁，请稍后再试」|
| 无效验证码场景 | scene 参数不合法 | 返回错误码 10015「无效的验证码场景」|
| Refresh Token 无效 | 刷新令牌不存在或已过期 | 返回错误码 10016「刷新令牌无效或已过期」|
| 原密码错误 | 修改密码时旧密码不匹配 | 返回错误码 10005「原密码不正确」|
| Access Token 过期 | Token 超过 2 小时 | 返回 HTTP 401，前端自动刷新 |
| Refresh Token 过期 | Token 超过 7 天 | 返回 HTTP 401，前端跳转登录页 |
| 手机号格式错误 | 非中国大陆手机号格式 | 参数校验失败，返回 400 |
